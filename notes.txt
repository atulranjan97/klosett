##################### Zod validation and type inference #######################################
We've been using .ts and .tsx file, we're using TS but we haven't utilized it to it's full potential.
for example, uptil now in our components -> shared -> `product-card.tsx` component we use the `any` type for product, which we shouldn't do, we should have a product type like 
    const ProductCard = ({ product }: {product: Product}) => {...}
So we're gonna create types, but we're also going to use a library called `Zod`.
so, Zod is a TS-first schema validation with static type inference. so not only does it cover types, it also offers runtime validation
If you just use TS on its own and you create your types or interfaces, that's great for development.
It offers compile time checking of your types, but it doesn't offer runtime validation for data from like APIs or forms. So, what this does is it allows us to create validation schemas so that we don't have to manually put validation like in our component or action that says like the name should have atleast three characters. Like we don't wanna clutter our code up with that, so we can just isolate it all inside of one file, inside of what's called a `Zod schema`.
And we can have schema for inserting products, for fetching products, deleting products or users or orders or whatever the resource might be

Let's start off by creating our types file. So, basically what we're gonna have is a validator file for our Zod schema. But ofcourse we wanna have our types file for our actual TS types
create `types`(folder in root) -> index.ts(this is where all our types will go)

Now we're not gonna add everything here because again we're using Zod and we can actually infer from our Zod schemas and that's what we're gonna do.

    import { z } from 'zod';

    // first schema that we wanna create is gonna be for inserting products, because again just the product on its own has fields like id, createdAt and rating, but those aren't fields we'd use in an insert, you're not gonna insert the rating when you add a product, you're not gonna insert the createdAt because that's created automatically in the DB

    // you might have different schema for updating the products

    // Schema for inserting products
    export const insertProductSchema = z.object({
      name: z.string().min(3, 'Name must be atleat 3 characters'),
      slug: z.string().min(3, 'Slug must be atleat 3 characters'),
      category: z.string().min(3, 'Category must be atleat 3 characters'),
      brand: z.string().min(3, 'Brand must be atleat 3 characters'),
      description: z.string().min(3, 'Description must be atleat 3 characters'),
      stock: z.coerce.number(),
      images: z.array(z.string()).min(1, 'Product must have atleat one image'),
      isFeatured: z.boolean(),
      banner: z.string().nullable(),
    })
    // this is very convenient because we're not gonna have to add this validation ourselves later on within actions or whatever we're submitting forms
    // we wanted `stock` to be a number but it's gonna come in as a string, most likely from a form. what we do is we'll write `z.coerce.number()`, so it'll coerce it to a number.
    // banner: z.string().nullable(), banner is gonna be a string but its going to be optional so we're gonna add on to that with `nullable()`
    // there's one more field I wanna add which is the price, but price need to formatted with precision because it needs to have two decimal points, something like 49.99, we don't want 49.9 or 49.99999. So, it's really important to format it correctly because it deals with taxes and totals and all that stuff so you really to make sure your price is formatted correctly.

तुम्हारे project में तुम Prisma ORM का use करके PostgreSQL database से `Product` data fetch कर रहे हो। जब Prisma database से data लाता है, तब वह normal JavaScript object जैसा दिखाई देता है, लेकिन internally कुछ fields special Prisma objects होती हैं। तुम्हारे `Product` schema में `price` और `rating` fields `Decimal` type की हैं, इसलिए Prisma इन्हें normal JavaScript `number` के रूप में नहीं बल्कि `Prisma.Decimal` object के रूप में return करता है। Prisma ऐसा इसलिए करता है क्योंकि JavaScript decimal calculations में precision errors आ सकते हैं, जैसे `0.1 + 0.2 = 0.30000000000000004`। Financial values और ratings जैसी decimal values को accurately handle करने के लिए Prisma special Decimal object use करता है ताकि exact precision बनी रहे।

Backend के अंदर यह Decimal object ठीक से काम करता है और Prisma इसके methods भी देता है, लेकिन problem तब आती है जब data को frontend, API response, Next.js Server Components, Client Components, server actions, या `JSON.stringify()` के through भेजना होता है। Next.js और JSON serialization केवल plain serializable values समझते हैं जैसे string, number, boolean, array और plain object। लेकिन Prisma Decimal एक custom class/object होता है, इसलिए कभी-कभी serialization errors, hydration issues या weird object structures दिखाई दे सकते हैं। इसी problem को avoid करने के लिए `convertToPlainObject()` function बनाया जाता है।

```ts id="7t1xk5"
export function convertToPlainObject<T>(value: T): T {
  return JSON.parse(JSON.stringify(value));
}
```

इस function में पहले `JSON.stringify(value)` Prisma result को JSON string में convert करता है। इस step में Prisma के special internal objects simple serializable values में बदल जाते हैं। उसके बाद `JSON.parse()` उस JSON string को वापस plain JavaScript object में convert कर देता है। इसका मतलब final output में Prisma internals नहीं रहते और object safely frontend या Next.js components में भेजा जा सकता है। उदाहरण के लिए `Decimal("999.99")` convert होकर `"999.99"` बन जाता है जो plain serializable value है।

इस function में `<T>` TypeScript generic है। इसका मतलब function किसी भी type का data accept कर सकता है और वही type return करेगा। अगर input `Product[]` है तो output भी `Product[]` माना जाएगा। इससे TypeScript type safety बनी रहती है और autocomplete भी सही काम करता है।

तुम्हारे specific case में conversion mainly इसलिए जरूरी है क्योंकि schema में `price Decimal` और `rating Decimal` fields मौजूद हैं। बाकी fields जैसे `String`, `Int`, `Boolean`, `String[]` आदि पहले से plain JavaScript compatible हैं और उन्हें convert करने की जरूरत नहीं होती। `createdAt` field `DateTime` है जो JavaScript `Date` object के रूप में आता है और अधिकतर cases में Next.js इसे automatically serialize कर देता है, इसलिए उसके लिए usually extra handling की जरूरत नहीं पड़ती।

Practical rule यह है कि अगर Prisma data केवल backend logic में use हो रहा है तो conversion optional है। लेकिन अगर वही data frontend, React components, API response, server action return value, cache, localStorage या network response में जा रहा है, तो उसे plain serializable object में convert करना recommended और safer approach है। इसलिए तुम्हारे Next.js + Prisma project में `convertToPlainObject()` use करना एक common और practical pattern है।
